# Internacionalizacion (i18n) y Localizacion (l10n)

Iulita soporta 6 idiomas en todas las capas: mensajes del sistema del backend, consola TUI, salidas de habilidades, vocabulario de aprobacion y el frontend del panel de control Vue 3.

## Idiomas Soportados

| Codigo | Idioma | RTL | Escritura |
|--------|--------|-----|-----------|
| `en` | Ingles | No | Latino |
| `ru` | Ruso | No | Cirilico |
| `zh` | Chino (Mandarin) | No | Hanzi |
| `es` | Espanol | No | Latino |
| `fr` | Frances | No | Latino |
| `he` | Hebreo | Si | Hebreo |

## Arquitectura

```
Backend (Go)                          Frontend (Vue 3)

│ ├── Bundle          │              │ ├── JSON locales    │
```

## i18n del Backend

### Estructura del Paquete

```
internal/i18n/
├── i18n.go        # Bundle, Init(), T(), Tl(), Translate()
├── context.go     # WithLocale(), LocaleFrom()
├── resolve.go     # ResolveLocale(), SupportedLanguages, matcher
└── catalog/
    ├── en.toml
    ├── ru.toml
    ├── zh.toml
    ├── es.toml
    ├── fr.toml
    └── he.toml
```

### Inicializacion

`i18n.Init()` debe llamarse despues de las migraciones de BD y antes del asistente/habilidades. Realiza:

1. Crea un bundle `go-i18n/v2` con ingles como idioma predeterminado
2. Registra el deserializador TOML
3. Lee todos los catalogos embebidos `*.toml` de `catalog/`
4. Pre-cachea instancias `*i18n.Localizer` para todos los idiomas soportados
5. Parsea las palabras de aprobacion en tablas de busqueda O(1)

### API de Traduccion

```go
// Basado en contexto (lee el locale del contexto)
i18n.T(ctx, "ConsolePlaceholder")
i18n.T(ctx, "WeatherForTitle", map[string]interface{}{"Location": "Moscow"})

// Directo basado en tag (sin contexto)
i18n.Tl(language.Russian, "LocaleSwitched", map[string]interface{}{"Language": "Russian"})

// Funcion de traduccion principal
i18n.Translate(tag, msgID, data...)
```

Cadena de respaldo: traduccion especifica del locale → traduccion al ingles → msgID sin procesar.

### Propagacion de Contexto

```go
// Establecer locale en el contexto
ctx = i18n.WithLocale(ctx, language.Russian)

// Leer locale del contexto
tag := i18n.LocaleFrom(ctx) // devuelve language.English si no esta establecido
```

El locale fluye a traves de todo el ciclo de vida de la solicitud:

```
user_channels.locale (BD)
    ↓
msg.Locale (campo IncomingMessage)
    ↓
i18n.ResolveLocale(msg.Locale, msg.LanguageCode)
    ↓
i18n.WithLocale(ctx, tag)
    ↓
Todas las llamadas i18n.T(ctx, ...) posteriores
```

### Resolucion de Locale

`ResolveLocale(channelLocale, msgLang)`:

1. Parsear `channelLocale` (preferencia almacenada en BD, ej., `"ru"`)
2. Recurrir a `msgLang` (idioma de la plataforma, ej., `LanguageCode` de Telegram)
3. Ejecutar a traves de `language.NewMatcher(SupportedLanguages)` para la mejor coincidencia
4. **Prioridad**: locale almacenado en BD > idioma de la plataforma > ingles predeterminado

### Palabras de Aprobacion

Cada locale define palabras afirmativas y negativas para el flujo de aprobacion:

```toml
# ru.toml
[ApprovalAffirmative]
other = "да,д,ок,подтвердить,подтверждаю,yes,y,ok,confirm"

[ApprovalNegative]
other = "нет,н,отмена,отменить,no,n,cancel"
```

**Garantia entre idiomas**: Todos los locales no ingleses incluyen palabras en ingles como respaldo. Los usuarios pueden escribir "yes" independientemente de su configuracion de locale.

La verificacion de aprobacion es O(1) usando conjuntos de palabras pre-parseados (`map[string]map[string]bool`).

### Directiva de Idioma del LLM

Cuando el locale no es ingles, se inyecta una directiva de idioma en el prompt dinamico del sistema:

```
# Ruso:
"Отвечай на русском языке. Язык ответов должен соответствовать предпочтениям пользователя."

# Hebreo:
"השב בעברית. שפת התשובות חייבת להתאים להעדפת המשתמש."
```

La directiva se renderiza en el idioma objetivo para que el LLM vea instrucciones que coinciden con su idioma de respuesta esperado. Los encabezados de seccion (`## Current Time`, `## Remembered Facts`) permanecen en ingles — el LLM los entiende independientemente.

### Notas del Catalogo TOML

- **Comillas chinas**: Usar corchetes `「」` en lugar de `""` en `zh.toml` — el parser TOML no puede manejar comillas de ancho completo dentro de cadenas regulares
- **Prompt de compresion**: Se mantiene en ingles en todos los locales — el LLM interpreta bien las instrucciones de resumen en ingles
- **Prefijo de resumen**: Traducido (aparece directamente en el chat)

## Almacenamiento de Locale por Canal

El locale se almacena por canal, no por usuario:

```sql
user_channels (
    ...
    locale TEXT NOT NULL DEFAULT 'en'
)
```

Esto permite diferentes idiomas en diferentes canales (ej., ruso en Telegram, ingles en WebChat).

### Operaciones de Almacenamiento

- `UpdateChannelLocale(ctx, chatID, locale)` — llamado por la habilidad `set_language`
- `GetChannelLocale(ctx, channelType, channelUserID)` — llamado durante el procesamiento de mensajes
- `GetChannelLocaleByChatID(ctx, chatID)` — busqueda alternativa

### Habilidad set_language

Los usuarios cambian de idioma via chat:

```
Usuario: "switch to Russian"
Bot: "Язык интерфейса изменён на русский."  (en ruso)

Usuario: "переключи на иврит"
Bot: "שפת הממשק שונתה לעברית."  (en hebreo)
```

La habilidad:
1. Parsea la entrada de idioma (codigo BCP-47 o nombre en ingles/ruso)
2. Valida contra los idiomas soportados
3. Actualiza `user_channels.locale` en la BD
4. Devuelve confirmacion en el **nuevo** idioma

## i18n del Frontend

### Configuracion

```typescript
// ui/src/i18n/index.ts
const i18n = createI18n({
    legacy: false,  // Composition API mode
    locale: localStorage.getItem('iulita.locale') || navigator.language.split('-')[0] || 'en',
    fallbackLocale: 'en',
    messages: { en, ru, zh, es, fr, he }
})
```

Los 6 archivos JSON de locale se importan estaticamente y se empaquetan en tiempo de compilacion.

### Composable useLocale

```typescript
// ui/src/composables/useLocale.ts
const { currentLocale, isRTL, direction, setLocale } = useLocale()
```

- `currentLocale` — ref reactivo, compartido entre todas las instancias del composable
- `isRTL` — computed: `true` para hebreo
- `direction` — computed: `'rtl'` o `'ltr'`
- `setLocale(tag)`:
  1. Actualiza el ref `currentLocale`
  2. Establece `locale.value` (vue-i18n)
  3. Guarda en `localStorage`
  4. Establece `document.documentElement.lang`
  5. `PATCH /api/auth/locale` con mejor esfuerzo para sincronizar el backend

### Soporte RTL

RTL se aplica por componente (no en `<html>`) para evitar romper el diseno del sidebar. Los componentes usan el valor computed `direction` para establecer atributos `dir` donde sea necesario.

### Componente LanguagePicker

Un dropdown `<NSelect>` de Naive UI en el sidebar con 6 opciones etiquetadas en sus escrituras nativas (ej., "עברית" para hebreo). Al seleccionar, llama a `setLocale()`.

### Estructura de Locale JSON

Objetos de espacio de nombres anidados:

```json
{
  "nav": { "dashboard": "Dashboard", "facts": "Facts", ... },
  "common": { "save": "Save", "cancel": "Cancel", ... },
  "login": { "title": "Login", ... },
  "dashboard": { ... },
  "tasks": { ... },
  "settings": { ... },
  "locale": { "en": "English", "ru": "Русский", "he": "עברית", ... }
}
```

### Sincronizacion de Locale con el Backend

`PATCH /api/auth/locale` (endpoint del panel de control):
1. Valida el locale via `i18n.IsSupported()`
2. Actualiza `user_channels.locale` para todos los canales del usuario
3. Devuelve el perfil de usuario actualizado

## Agregar un Nuevo Idioma

### Backend

1. Crear `internal/i18n/catalog/<code>.toml` con todas las claves requeridas
2. Agregar el `language.Tag` a `SupportedLanguages` en `resolve.go`
3. Agregar flag RTL si es necesario
4. Agregar mapeos de nombres de idioma en `skill/locale/locale.go` (nombres en ingles + ruso)

### Frontend

1. Crear `ui/src/i18n/locales/<code>.json` con todos los espacios de nombres
2. Importar y registrar en `ui/src/i18n/index.ts`
3. Agregar a RTL_LOCALES en `ui/src/composables/useLocale.ts` si es RTL
4. Agregar etiqueta nativa en todos los archivos de locale bajo `locale.<code>`
