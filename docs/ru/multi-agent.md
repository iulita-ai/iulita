# Мультиагентная оркестрация

Iulita поддерживает параллельное выполнение суб-агентов для декомпозиции сложных задач. LLM самостоятельно решает, когда использовать оркестрацию, исходя из сложности задачи.

## Обзор

Навык `orchestrate` запускает несколько специализированных суб-агентов параллельно. Каждый суб-агент выполняет упрощённый агентный цикл со своим системным промптом, подмножеством инструментов и опциональной маршрутизацией LLM-провайдера. Результаты собираются и возвращаются родительскому ассистенту в виде структурированного markdown-отчёта.

## Архитектура

```
Сообщение пользователя
    │
    ▼
Ассистент (основной агентный цикл)
    │ решает, что нужна оркестрация
    │ вызывает инструмент orchestrate со спецификациями агентов
    ▼
Навык Orchestrate
    │ проверяет глубину (максимум 1)
    │ создаёт Budget из конфигурации + входных параметров
    ▼
Оркестратор (параллельное выполнение через errgroup)
    ├── Runner (agent_1: researcher) ──→ LLM ←→ Инструменты
    ├── Runner (agent_2: analyst)    ──→ LLM ←→ Инструменты
    └── Runner (agent_3: planner)    ──→ LLM ←→ Инструменты
         │
         │ общий атомарный бюджет токенов
         │ тайм-аут + контекст для каждого агента
         │ события статуса → каналы
         ▼
    Собранные AgentResults
    │ форматирование в markdown
    ▼
Ассистент продолжает работу с результатами оркестрации
```

## Типы агентов

Каждый тип агента имеет специализированный системный промпт и опциональные инструменты по умолчанию.

| Тип | Фокус системного промпта | Инструменты по умолчанию | Route Hint |
|-----|--------------------------|--------------------------|------------|
| `researcher` | Сбор информации, поиск в интернете, структурированные сводки | `web_search`, `webfetch` | — |
| `analyst` | Выявление паттернов, аномалий, ключевых инсайтов | все | — |
| `planner` | Декомпозиция целей в упорядоченные шаги | `datetime` | — |
| `coder` | Написание, ревью, отладка кода | все | — |
| `summarizer` | Сжатие ввода до ключевых моментов | все | `ollama` |
| `generic` | Общего назначения | все | — |

## Использование

LLM автономно решает, когда применять оркестрацию. SKILL.md навыка `orchestrate` внедряется в системный промпт, указывая LLM, когда оркестрация уместна:

- Задачи, выигрывающие от параллельного исследования (исследование + анализ)
- Многокомпонентные задачи, где разные аспекты могут прорабатываться одновременно
- Задачи, требующие различной экспертизы

## Система бюджета

### Общий бюджет токенов

Все агенты разделяют единый атомарный счётчик `atomic.Int64`. Каждый вызов LLM агентом вычитает из общего пула.

**Мягкое ограничение**: несколько агентов могут одновременно пройти проверку бюджета до того, как кто-либо вычтет токены. Перерасход может составить до `(N_агентов - 1) * токенов_за_вызов`. Это сделано намеренно — жёсткое резервирование добавило бы конкуренцию мьютексов без значимой пользы.

### Параметры по умолчанию

| Параметр | По умолчанию | Переопределение |
|----------|-------------|-----------------|
| Максимум итераций (вызовов LLM) | 10 | `Budget.MaxTurns` |
| Тайм-аут (реальное время) | 60с | `Budget.Timeout` или входной `timeout` |
| Максимум параллельных агентов | 5 | `Budget.MaxAgents` или конфигурация |
| Общий бюджет токенов | неограничен | `Budget.MaxTokens` или входной `max_tokens` |

## Контроль глубины

Суб-агенты не могут порождать дальнейших суб-агентов. Это обеспечивается на двух уровнях:

1. **Ключ глубины в контексте**: `WithDepth(ctx, DepthFrom(ctx)+1)` устанавливается в контексте каждого суб-агента. Навык orchestrate проверяет `DepthFrom(ctx) >= MaxDepth`.

2. **Фильтрация инструментов**: `buildTools()` всегда исключает инструмент `orchestrate` из списка инструментов суб-агентов.

`MaxDepth = 1` означает: глубина 0 = родительский ассистент, глубина 1 = суб-агент (максимум).

## Безопасность

### Фильтрация подтверждений

Суб-агенты обходят обычный процесс подтверждения, поэтому они не должны иметь доступ к навыкам, требующим подтверждения:

- `shell_exec` (ApprovalManual) — недоступен
- Docker executor (ApprovalPrompt) — недоступен
- Любой навык с уровнем подтверждения выше ApprovalAuto — недоступен

### Белые списки инструментов

Каждая спецификация агента может опционально объявить явный белый список инструментов. Если не указан, используются инструменты по умолчанию для типа агента.

## Протокол событий статуса

| Событие | Когда | Поля данных |
|---------|-------|-------------|
| `orchestration_started` | Перед запуском агентов | `agent_count` |
| `agent_started` | Для каждого агента, перед запуском | `agent_id`, `agent_type` |
| `agent_progress` | Для каждого агента, после каждой итерации LLM | `agent_id`, `turn` |
| `agent_completed` | Для каждого агента, при успехе | `agent_id`, `tokens`, `duration_ms` |
| `agent_failed` | Для каждого агента, при ошибке | `agent_id`, `error` |
| `orchestration_done` | После завершения всех агентов | `success_count`, `total_tokens`, `duration_ms` |

События после завершения агентов используют `context.Background()` с тайм-аутом 5 секунд для гарантии доставки даже при истечении крайнего срока родительского контекста.

## Маршрутизация LLM-провайдеров

Каждый тип агента может иметь `RouteHint` для выбора провайдера:

1. **Переопределение на уровне спецификации**: `AgentSpec.RouteHint`
2. **Значение по умолчанию профиля**: `AgentTypeProfile.RouteHint` (напр., `summarizer → "ollama"`)
3. **Пусто**: использовать провайдер по умолчанию

## Конфигурация

| Ключ | По умолчанию | Описание |
|------|-------------|----------|
| `skills.orchestrate.enabled` | true | Включить/выключить навык orchestrate |
| `skills.orchestrate.max_tokens` | 0 (неограничено) | Лимит общего бюджета токенов |
| `skills.orchestrate.max_agents` | 5 | Максимум параллельных агентов |
| `skills.orchestrate.timeout` | 60s | Тайм-аут для каждого агента |
| `skills.orchestrate.request_timeout` | 1h | Общий дедлайн оркестрации (максимум 4h) |

Все ключи поддерживают горячую перезагрузку через `ConfigReloadable`.

## Фронтенд

Компонент `AgentProgress.vue` отображает статус агентов в реальном времени в представлении чата:

- Показывает активную оркестрацию с общим количеством агентов
- Строка для каждого агента с иконкой типа, именем, иконкой статуса и информацией о прогрессе
- Завершённые агенты показывают длительность; неудавшиеся — сокращённую ошибку
