# Каналы

Iulita поддерживает несколько каналов связи. Каждый канал преобразует платформо-специфичные сообщения в универсальный формат `IncomingMessage` и маршрутизирует их через ассистента.

## Возможности каналов

Каждый канал объявляет свои возможности через битовую маску на каждом сообщении:

| Возможность | Console | Telegram | WebChat |
|-------------|---------|----------|---------|
| Стриминг | через bubbletea | Да (через редактирование) | Да (WebSocket) |
| Markdown | через glamour | Да | HTML |
| Реакции | Нет | Нет | Нет |
| Кнопки | Нет | Да (inline-клавиатура) | Да |
| Индикатор набора | Да | Да | Нет |
| HTML | Нет | Нет | Да |

Возможности привязаны к сообщению (не к каналу), что позволяет различное поведение при работе нескольких каналов с одним ассистентом. Навыки могут проверять возможности через `channel.CapsFrom(ctx)` для адаптации формата вывода.

## Console TUI

Режим по умолчанию — полноэкранный терминальный чат на базе [bubbletea](https://github.com/charmbracelet/bubbletea).

### Возможности

- **Полноэкранная раскладка**: viewport (история чата) + разделитель + textarea (ввод) + строка статуса
- **Рендеринг Markdown**: через [glamour](https://github.com/charmbracelet/glamour) с адаптивным переносом слов
- **Стриминг**: появление текста в реальном времени с индикатором загрузки
- **Слэш-команды**: `/help`, `/status`, `/compact`, `/clear`, `/quit`
- **Интерактивные запросы**: нумерованные варианты для взаимодействия с навыками (напр., выбор локации погоды)
- **Определение фона**: адаптация рендеринга перед запуском bubbletea

### Архитектура

```
tuiModel (bubbletea)
    ├── viewport.Model (прокручиваемая история чата)
    ├── textarea.Model (ввод пользователя)
    ├── statusBar (название навыка, счётчики токенов, расходы)
    └── streamBuf (текст стриминга в реальном времени)
```

Структура `console.Channel` содержит `*tea.Program`, защищённый `sync.RWMutex`. Программа bubbletea работает в своей горутине (блокирующий `Start()`), в то время как `StartStream`, `SendMessage` и `NotifyStatus` вызываются из горутины ассистента конкурентно.

### Мост стриминга

Когда ассистент стримит ответ:

1. `StartStream()` возвращает замыкания `editFn` и `doneFn`
2. `editFn(text)` отправляет `streamChunkMsg` в bubbletea (накопленный полный текст)
3. `doneFn(text)` отправляет `streamDoneMsg` в bubbletea (финализация и добавление в историю)
4. Все сообщения потокобезопасны через `p.Send()` bubbletea

### Слэш-команды

| Команда | Описание |
|---------|----------|
| `/help` | Показать все команды с описаниями |
| `/status` | Количество навыков, дневные расходы, токены сессии, количество сообщений |
| `/compact` | Вручную запустить сжатие истории (асинхронно) |
| `/clear` | Очистить историю чата в памяти (только TUI) |
| `/quit` / `/exit` | Выйти из приложения |

### Совместная работа с серверным режимом

В консольном режиме сервер работает в фоне:
- Логи перенаправляются в `iulita.log` (не в stderr, чтобы не портить TUI)
- Дашборд по-прежнему доступен по настроенному адресу
- Telegram и другие каналы работают параллельно с TUI

## Telegram

Полнофункциональный Telegram-бот со стримингом, дебаунсингом и интерактивными запросами.

### Настройка

1. Создайте бота через [@BotFather](https://t.me/BotFather)
2. Задайте токен: `iulita init` (keyring) или переменная окружения `IULITA_TELEGRAM_TOKEN`
3. Опционально: задайте `telegram.allowed_ids` для ограничения доступа по ID пользователей Telegram

### Возможности

- **Белый список пользователей**: `allowed_ids` ограничивает доступ к боту. Пустой = разрешить всем (логируется предупреждение)
- **Дебаунсинг сообщений**: быстрые сообщения из одного чата объединяются (настраиваемое окно)
- **Стриминг через редактирование**: ответы появляются постепенно через `EditMessageText` (ограничение 1 редактирование/1.5 с)
- **Разбиение сообщений**: сообщения длиннее 4000 символов разбиваются по границам абзацев/строк/слов с сохранением блоков кода
- **Цепочка ответов**: первый фрагмент отвечает на сообщение пользователя; последующие — отдельные
- **Индикатор набора**: действие `ChatTyping` отправляется каждые 4 секунды во время обработки
- **Мониторинг здоровья**: `GetMe()` вызывается каждые 60 секунд для обнаружения проблем
- **Интерактивные запросы**: inline-клавиатуры для взаимодействия с навыками (погода и т.д.)
- **Поддержка медиа**: фото (наибольший размер), документы (лимит 30 МБ), голос/аудио (с транскрипцией)
- **Встроенные команды**: `/clear` (очистить историю), пользовательские зарегистрированные команды
- **Кнопка закладки**: inline-кнопка 💾 под каждым ответом ассистента; нажатие сохраняет полный ответ как факт с фоновым уточнением через LLM
- **Статусные сообщения в реальном времени**: обновления статуса во время выполнения инструментов и оркестрации агентов, показывающие текущий навык и прогресс агентов

### Конвейер обработки сообщений

```
Входящее обновление Telegram
    │
    ├── Callback query?
    │   ├── "noop" → подтверждение без действия
    │   ├── "remember:*" → обработчик закладок (сохранение факта + обратная связь ✅)
    │   └── прочие → перенаправление обработчику запросов
    ├── Не сообщение? → пропуск
    ├── Пользователь не в белом списке? → отклонение
    ├── Команда /clear? → прямая обработка
    ├── Зарегистрированная команда? → перенаправление обработчику
    ├── Активный запрос? → перенаправление текста в запрос
    │
    ▼
Создание IncomingMessage
    │ Caps = Streaming | Markdown | Typing | Buttons
    │
    ├── Разрешение пользователя (платформа → iulita UUID)
    ├── Определение локали из БД
    ├── Загрузка медиа (фото/документ/голос)
    ├── Проверка ограничения частоты
    │
    ▼
Дебаунсинг
    │ объединение быстрых сообщений (текст через \n)
    │ таймер сбрасывается при каждом новом сообщении
    │
    ▼
Обработчик (Assistant.HandleMessage)
```

### Дебаунсер

Дебаунсер объединяет быстрые сообщения из одного чата для предотвращения множественных вызовов LLM:

- У каждого `chatID` есть буфер с таймером `time.AfterFunc`
- Добавление сообщения сбрасывает таймер
- При срабатывании таймера все буферизованные сообщения объединяются:
  - Текст соединяется через `"\n"`
  - Изображения и документы объединяются
  - Метаданные первого сообщения сохраняются
- Если `debounce_window = 0`, сообщения обрабатываются немедленно (неблокирующе)
- `flushAll()` обрабатывает оставшиеся буферы при завершении

### Разбиение сообщений

Длинные ответы разбиваются на фрагменты, совместимые с Telegram (максимум 4000 символов):

1. Попытка разбиения по границам абзацев (`\n\n`)
2. Попытка разбиения по границам строк (`\n`)
3. Попытка разбиения по границам слов (` `)
4. Жёсткое разбиение как крайний случай
5. **Учёт блоков кода**: при разбиении внутри блока ``` он закрывается ``` и открывается заново в следующем фрагменте

### Конфигурация

| Ключ | По умолчанию | Описание |
|------|-------------|----------|
| `telegram.token` | — | Токен бота (горячая перезагрузка) |
| `telegram.allowed_ids` | `[]` | Белый список ID пользователей (пустой = все) |
| `telegram.debounce_window` | 2s | Окно объединения сообщений |

## WebChat

Веб-чат на базе WebSocket, встроенный в дашборд.

### Протокол

**Подключение**: WebSocket по `/ws/chat?user_id=<uuid>&username=<name>&chat_id=<optional>`

**Входящие сообщения** (клиент -> сервер):
```json
{
  "text": "user message",
  "chat_id": "web:abc123",
  "prompt_id": "prompt_123_1",       // только для ответов на запросы
  "prompt_answer": "option_id",      // только для ответов на запросы
  "remember_message_id": "nano_ts"   // только для запросов закладок
}
```

**Исходящие сообщения** (сервер -> клиент):

| Тип | Назначение | Ключевые поля |
|-----|-----------|---------------|
| `message` | Обычный ответ | `text`, `timestamp` |
| `stream_edit` | Обновление стриминга | `text`, `message_id`, `timestamp` |
| `stream_done` | Завершение стриминга | `text`, `message_id`, `timestamp` |
| `status` | События обработки | `status`, `skill_name`, `success`, `duration_ms` |
| `prompt` | Интерактивный вопрос | `text`, `prompt_id`, `options[]` |
| `remember_ack` | Подтверждение закладки | `remember_ack.message_id`, `remember_ack.fact_id`, `remember_ack.status` |

### Протокол закладок

Функция закладок позволяет пользователям сохранять ответы ассистента как факты через кнопку в интерфейсе.

**Процесс:**
1. Сервер отправляет `message` или `stream_done` с `message_id` (Unix наносекундная метка)
2. Сервер кэширует содержимое по ключу `(message_id, chatID)` на 10 минут
3. Фронтенд показывает иконку 💾 при наведении на сообщения ассистента
4. Пользователь нажимает → отправляется `{"remember_message_id": "<message_id>"}`
5. Сервер проверяет владельца (`chatID` должен совпадать с кэшированной записью), сохраняет как факт с `source_type="bookmark"`, ставит в очередь фоновое уточнение через LLM
6. Сервер отправляет `{"type": "remember_ack", "remember_ack": {"message_id": "...", "status": "saved", "fact_id": 42}}`
7. Фронтенд обновляет иконку на ✅

**Значения status**: `saved`, `error`, `expired` (сообщение больше не в кэше)

### Аутентификация

WebChat **не** использует UserResolver. Фронтенд получает JWT-токен через `/api/auth/login`, извлекает `user_id` из payload и передаёт его как параметр WebSocket-запроса. Канал доверяет этому `user_id` напрямую.

### Сериализация записи

Все записи WebSocket проходят через per-connection `sync.Mutex` для предотвращения паники при конкурентной записи. Каждое соединение отслеживается в карте `clients[chatID]`.

### Интерактивные запросы

Запросы используют ID на основе атомарного счётчика: `prompt_<timestamp>_<counter>`. Сервер отправляет сообщение `prompt` с вариантами; клиент отвечает с `prompt_id` и `prompt_answer`. Ожидающие запросы хранятся в `sync.Map` с тайм-аутом.

## Менеджер каналов

`channelmgr.Manager` оркестрирует все экземпляры каналов во время выполнения.

### Жизненный цикл

- **StartAll**: загружает все экземпляры каналов из БД, запускает каждый в горутине
- **StopInstance**: отменяет контекст, ждёт done-канал (тайм-аут 5 с)
- **AddInstance / UpdateInstance**: для экземпляров, созданных/изменённых через дашборд
- **Горячая перезагрузка**: `UpdateConfigToken(token)` перезапускает экземпляры Telegram из конфигурации

### Маршрутизация сообщений

Когда ассистенту нужно отправить проактивное сообщение (напоминание, heartbeat):

1. Поиск экземпляра канала, владеющего `chatID`, через БД
2. Если найден и запущен, используется его sender
3. Резерв: использование первого запущенного канала

### Поддерживаемые типы каналов

| Тип | Источник | Горячая перезагрузка |
|-----|----------|---------------------|
| Telegram | Конфигурация или БД | Горячая перезагрузка токена |
| WebChat | БД (bootstrap) | — |
| Console | Только консольный режим | — |

## Разрешение пользователей

`DBUserResolver` сопоставляет идентификаторы платформ с UUID iulita:

1. Поиск в `user_channels` по `(channel_type, channel_user_id)`
2. Если найден -> возврат существующего `user.ID`
3. Если не найден и авто-регистрация включена:
   - Создание нового `User` со случайным паролем и `MustChangePass: true`
   - Привязка канала к пользователю
   - Возврат нового UUID
4. Если не найден и авто-регистрация отключена -> отклонение

**Локаль по каналу**: после разрешения вызывается `GetChannelLocale(ctx, channelType, channelUserID)` для установки `msg.Locale` из сохранённого в БД предпочтения.

## События статуса

Каналы получают уведомления `StatusEvent` для обратной связи UX:

| Тип | Когда | Использование |
|-----|-------|--------------|
| `processing` | Сообщение получено, до вызова LLM | Показать "думаю..." |
| `skill_start` | Перед выполнением навыка | Показать название навыка |
| `skill_done` | После выполнения навыка | Показать длительность, успех/неудачу |
| `stream_start` | Перед началом стриминга | Подготовить UI стриминга |
| `error` | При ошибке | Показать сообщение об ошибке |
| `locale_changed` | После навыка set_language | Обновить локаль UI |
| `orchestration_started` | Перед запуском суб-агентов | Показать количество агентов |
| `agent_started` | Для каждого агента, перед запуском | Показать имя + тип агента |
| `agent_progress` | Для каждого агента, после каждой итерации LLM | Обновить счётчик итераций |
| `agent_completed` | Для каждого агента, при успехе | Показать длительность + токены |
| `agent_failed` | Для каждого агента, при ошибке | Показать сообщение об ошибке |
| `orchestration_done` | После завершения всех агентов | Показать итоговую статистику |
