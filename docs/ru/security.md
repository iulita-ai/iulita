# Безопасность

## Аутентификация

### JWT

- **Алгоритм**: HMAC-SHA256
- **TTL access-токена**: 24 часа
- **TTL refresh-токена**: 7 дней
- **Claims**: `user_id`, `username`, `role`
- **Секрет**: автогенерируемый 32-байтовый hex, если не задан
- **Хранение**: хранилище ключей (macOS Keychain / Linux SecretService) или переменная `IULITA_JWT_SECRET`

### Пароли

- **Хеширование**: bcrypt со стандартной стоимостью
- **Bootstrap**: первый пользователь получает случайный пароль с `MustChangePass: true`
- **Дашборд**: смена пароля через эндпоинт `POST /api/auth/change-password`

### Middleware

1. `FiberMiddleware` — валидирует `Authorization: Bearer <token>`, сохраняет claims в fiber locals
2. `AdminOnly` — проверяет `role == admin`, возвращает 403 если нет

## Управление секретами

### Слои хранения

| Секрет | Переменная окружения | Хранилище ключей | Файловый резерв |
|--------|---------------------|------------------|-----------------|
| API-ключ Claude | `IULITA_CLAUDE_API_KEY` | `claude-api-key` | — |
| Токен Telegram | `IULITA_TELEGRAM_TOKEN` | `telegram-token` | — |
| Секрет JWT | `IULITA_JWT_SECRET` | `jwt-secret` | автогенерация |
| Ключ шифрования конфигурации | `IULITA_CONFIG_KEY` | `config-encryption-key` | файл `encryption.key` |

**Порядок разрешения**: переменная окружения -> хранилище ключей -> файловый резерв -> автогенерация.

### Шифрование конфигурации (AES-256-GCM)

Переопределения конфигурации в рантайме, хранящиеся в базе данных, могут быть зашифрованы:

- **Алгоритм**: AES-256-GCM (аутентифицированное шифрование)
- **Nonce**: 12 байт, генерируется случайно для каждого шифрования
- **Формат**: `base64(nonce ‖ шифротекст)`
- **Авто-шифрование**: ключи, объявленные как `secret_keys` в манифестах SKILL.md
- **Безопасность API**: дашборд никогда не возвращает расшифрованные значения для секретных ключей
- **Отклонение placeholder**: значения `"***"` или пустые значения отклоняются для секретных ключей

## Защита от SSRF

Все исходящие HTTP-запросы (web fetch, веб-поиск, внешние навыки) проходят через защиту от SSRF.

### Заблокированные диапазоны IP

| Диапазон | Тип |
|----------|-----|
| `10.0.0.0/8` | RFC1918 приватный |
| `172.16.0.0/12` | RFC1918 приватный |
| `192.168.0.0/16` | RFC1918 приватный |
| `100.64.0.0/10` | Carrier-grade NAT (RFC 6598) |
| `fc00::/7` | IPv6 Unique Local |
| `127.0.0.0/8` | Loopback |
| `::1/128` | IPv6 loopback |
| `169.254.0.0/16` | Link-local |
| `fe80::/10` | IPv6 link-local |
| Multicast-диапазоны | Все |

IPv4-mapped IPv6-адреса нормализуются к IPv4 перед проверкой.

### Двухуровневая защита (без прокси)

**Уровень 1 — Pre-flight DNS**: Перед подключением все IP для hostname разрешаются. Если любой IP является приватным, соединение отклоняется.

**Уровень 2 — Connect-time Control**: Функция `net.Dialer.Control` проверяет фактический разрешённый IP во время подключения. Это ловит **атаки DNS rebinding**, когда hostname разрешается в публичный IP при pre-flight, но перепривязывается к приватному IP перед фактическим подключением.

### Путь через прокси

При настроенном прокси (`proxy.url`) подход на основе dialer не работает (сам прокси может иметь приватный IP в Kubernetes-кластерах). Вместо этого:

- `ssrfTransport.RoundTrip` выполняет только pre-flight проверку на уровне URL
- Подключение прокси к приватным IP разрешено (намеренно для кластерно-внутренних прокси)
- Целевые URL к приватным IP по-прежнему заблокированы

### Активное обнаружение прокси

`isProxyActive()` фактически вызывает функцию прокси с тестовым запросом (не просто `Proxy != nil`), потому что `http.DefaultTransport` всегда имеет `Proxy = ProxyFromEnvironment`.

## Уровни подтверждения инструментов

| Уровень | Поведение | Навыки |
|---------|----------|--------|
| `ApprovalAuto` | Немедленное выполнение | Большинство навыков (по умолчанию) |
| `ApprovalPrompt` | Пользователь должен подтвердить | Docker executor |
| `ApprovalManual` | Администратор должен подтвердить | Shell exec |

### Процесс

1. Навык объявляет свой уровень через интерфейс `ApprovalDeclarer`
2. Перед выполнением ассистент проверяет `registry.ApprovalLevelFor(toolName)`
3. Для `Prompt`/`Manual`: сохраняет ожидающий вызов инструмента в `approvalStore`
4. Отправляет запрос подтверждения пользователю
5. Возвращает `"awaiting approval"` в LLM (неблокирующе)
6. Следующее сообщение пользователя проверяется по словарю подтверждения с учётом локали
7. При одобрении: выполняет сохранённый вызов инструмента, возвращает результат
8. При отклонении: возвращает "cancelled"

### Словарь с учётом локали

Слова подтверждения определены во всех 6 языковых каталогах и включают английский как резерв:

```
# Русские утвердительные:
да, д, ок, подтвердить, подтверждаю, yes, y, ok, confirm

# Ивритские отрицательные:
לא, ביטול, בטל, no, n, cancel
```

## Безопасность Telegram

- **Белый список пользователей**: `telegram.allowed_ids` ограничивает доступ к боту
- **Пустой белый список**: разрешает всех пользователей (логируется предупреждение)
- **Ограничение частоты**: скользящее окно rate limiter для каждого чата

## Безопасность Shell Exec

Навык `shell_exec` имеет самый строгий уровень безопасности:

- **Уровень подтверждения**: `ApprovalManual` (требуется подтверждение администратора)
- **Только по белому списку**: только исполняемые файлы из `AllowedBins`
- **Запрещённые пути**: настраиваемый список путей, которые не могут появляться в аргументах
- **Обход пути**: `..` в аргументах отклоняется
- **Лимит вывода**: максимум 16 КБ
- **Рабочая директория**: `os.TempDir()` (не директория проекта)

## Ограничение частоты запросов

### Per-Chat Rate Limiter

Скользящее окно: отслеживает метки времени по `chatID`. Если количество сообщений в пределах `window` превышает `rate`, сообщение отклоняется.

### Глобальный Action Limiter

Фиксированное окно: общее количество LLM/tool-действий в час по всем чатам. Авто-сброс на границе окна.

## Отслеживание расходов

- **В памяти**: дневные расходы с mutex, авто-сброс на границе дня
- **Персистентно**: `IncrementUsageWithCost` сохраняет в таблицу `usage_stats`
- **Дневной лимит**: `cost.daily_limit_usd` (0 = без ограничений)
- **Цены по моделям**: `config.ModelPrice{InputPerMillion, OutputPerMillion}`

## CI/CD-безопасность

- **Pre-commit hook**: блокирует секреты через [gitleaks](https://github.com/gitleaks/gitleaks)
- **CI**: gitleaks action сканирует все коммиты
- **CodeQL**: security-extended запросы для Go и JavaScript/TypeScript (когда репозиторий публичный)
- **Зависимости**: оповещения Dependabot (включить в настройках GitHub)

## Безопасность внешних навыков

- **Валидация slug**: `^[a-zA-Z0-9][a-zA-Z0-9_-]{0,63}$` — предотвращает обход пути
- **Проверка контрольной суммы**: SHA-256 для удалённых загрузок
- **Валидация изоляции**: навыки должны объявлять уровень изоляции, проверяется по флагам конфигурации
- **Обнаружение кода**: отклоняет навыки с файлами кода, если не изолированы должным образом
- **Сканирование prompt injection**: предупреждает о подозрительных паттернах в теле навыка
- **Максимальный размер архива**: настраиваемый (по умолчанию 50 МБ для ClawhHub)
