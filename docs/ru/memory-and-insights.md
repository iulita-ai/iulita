# Память и инсайты

## Философия

Большинство AI-ассистентов либо забывают всё между сессиями, либо галлюцинируют «воспоминания» из обучающих данных. Iulita использует принципиально иной подход:

- **Только явное сохранение** — факты сохраняются только когда вы просите ("запомни, что...")
- **Проверенные данные** — каждый факт восходит к конкретному запросу пользователя
- **Перекрёстные инсайты** — паттерны обнаруживаются путём анализа ваших реальных фактов
- **Временное затухание** — старые воспоминания естественно теряют значимость, если к ним не обращаются
- **Гибридный поиск** — полнотекстовый FTS5 в сочетании с ONNX-векторными эмбеддингами

## Факты

### Сохранение фактов (Remember)

Когда вы говорите "запомни, что моего пса зовут Макс", происходит следующее:

1. **Определение триггера** — ассистент обнаруживает ключевое слово памяти ("запомни") и принудительно вызывает инструмент `remember`
2. **Проверка дубликатов** — поиск среди существующих фактов по первым 3 словам для обнаружения почти-дубликатов
3. **SQLite INSERT** — факт сохраняется с `user_id`, `content`, `source_type=user`, метками времени
4. **Индекс FTS5** — триггер `AFTER INSERT` автоматически добавляет факт в полнотекстовый индекс `facts_fts`
5. **Векторный эмбеддинг** — фоновая горутина генерирует ONNX-эмбеддинг (384-мерный all-MiniLM-L6-v2) и сохраняет в `fact_vectors`

```
domain.Fact {
    ID             int64
    ChatID         string     // исходный канал ("123456789", "console", "web:uuid")
    UserID         string     // iulita UUID — общий для всех каналов
    Content        string     // текст факта
    SourceType     string     // "user" (явный) или "system" (автоизвлечённый)
    CreatedAt      time.Time
    LastAccessedAt time.Time  // сбрасывается при каждом извлечении
    AccessCount    int        // увеличивается при каждом извлечении
}
```

### Извлечение фактов (Recall)

Когда вы спрашиваете "как зовут моего пса?":

1. **Поиск по пользователю** — сначала пробует `SearchFactsByUser(userID, query, limit)` для кроссканальных фактов
2. **FTS5-сопоставление** — `SELECT * FROM facts WHERE id IN (SELECT rowid FROM facts_fts WHERE facts_fts MATCH ?)`
3. **Избыточная выборка** — загружает `limit * 3` кандидатов (минимум 20) для последующего переранжирования
4. **Временное затухание** — каждый кандидат оценивается: `decay = exp(-ln(2) / halfLife * ageDays) * (1 + log(1 + accessCount))`
5. **MMR-переранжирование** — Maximal Marginal Relevance снижает число почти-дубликатов в результатах
6. **Усиление** — у каждого возвращённого факта обновляется `access_count++` и `last_accessed_at = now`

Ассистент также выполняет **гибридный поиск** при каждом сообщении (не только при явном извлечении):

```
1. Очистка запроса (удаление FTS-операторов, стоп-слов, ограничение до 5 ключевых слов)
2. Генерация вектора запроса через ONNX-эмбеддинг
3. FTS-результаты: оценка по позиции (1 - i/(n+1))
4. Векторные результаты: косинусное сходство со всеми сохранёнными векторами
5. Комбинированная: (1-vectorWeight)*ftsScore + vectorWeight*vecScore
6. Объединение обоих наборов, сортировка по комбинированной оценке
```

### Удаление фактов

Инструмент `forget` удаляет факт по ID. FTS-триггер (`facts_ad`) автоматически удаляет его из полнотекстового индекса. `ON DELETE CASCADE` на `fact_vectors` удаляет эмбеддинг.

## Временное затухание

Факты и инсайты затухают со временем по экспоненциальному (радиоактивному) закону:

```
decay_factor = exp(-ln(2) / halfLifeDays * ageDays)
```

| Дней с последнего обращения | Период полураспада = 30 дней | Период полураспада = 90 дней |
|-----------------------------|-----------------------------|-----------------------------|
| 0 | 1.00 | 1.00 |
| 15 | 0.71 | 0.89 |
| 30 | 0.50 | 0.79 |
| 60 | 0.25 | 0.63 |
| 90 | 0.13 | 0.50 |

**Ключевые решения:**
- **Факты** затухают от `last_accessed_at` — каждое извлечение сбрасывает таймер
- **Инсайты** затухают от `created_at` — у них фиксированное время жизни
- **Бонус за обращения**: `1 + log(1 + accessCount)` — факт, к которому обращались 100 раз, получает 4.6-кратный бонус
- **Период полураспада по умолчанию**: 30 дней (настраивается через `skills.memory.half_life_days`)

## MMR-переранжирование

После оценки временного затухания Maximal Marginal Relevance предотвращает почти-дубликаты в результатах:

```
MMR(item) = lambda * relevance_score - (1 - lambda) * max_similarity_to_selected
```

- `lambda = 1.0` — чистая релевантность, без разнообразия
- `lambda = 0.7` — рекомендуется: приоритет релевантности, но штраф за почти-дубликаты
- `lambda = 0.0` — чистое разнообразие

Схожесть измеряется через сходство Жаккара на токенах слов (приближение без зависимостей, работает без ONNX-эмбеддера).

**Конфигурация**: `skills.memory.mmr_lambda` (по умолчанию 0, отключено). Установите 0.7 для лучших результатов.

## Инсайты

Инсайты — это AI-генерируемые перекрёстные ссылки между вашими фактами, обнаруживаемые фоновым планировщиком.

### Конвейер генерации

Задача генерации инсайтов запускается каждые 24 часа (настраивается):

```
1. Загрузка всех фактов пользователя
2. Проверка минимального количества фактов (по умолчанию 20)
3. Построение TF-IDF-векторов
   - Токенизация: нижний регистр, удаление пунктуации, фильтрация стоп-слов
   - Генерация биграмм (пары соседних слов)
   - Вычисление TF-IDF-оценок
4. K-means++ кластеризация
   - k = sqrt(numFacts / 3)
   - Метрика косинусного расстояния
   - Максимум 20 итераций
5. Выборка межкластерных пар
   - До 6 пар за запуск
   - Пропуск уже обработанных пар фактов
6. Для каждой пары:
   a. Отправка в LLM: "Сгенерируй творческий инсайт из этих двух кластеров"
   b. Оценка качества (1-5) отдельным вызовом LLM
   c. Сохранение, если качество >= порога
```

### Жизненный цикл инсайта

```
domain.Insight {
    ID             int64
    ChatID         string
    UserID         string
    Content        string     // текст инсайта
    FactIDs        string     // ID исходных фактов через запятую
    Quality        int        // оценка качества LLM 1-5
    AccessCount    int
    LastAccessedAt time.Time
    CreatedAt      time.Time
    ExpiresAt      *time.Time // по умолчанию: created + 30 дней
}
```

- **Создаётся** фоновым планировщиком после кластеризации и синтеза LLM
- **Отображается** в системном промпте ассистента при контекстуальной релевантности (гибридный поиск)
- **Усиливается** при обращении (обновляются счётчик обращений и время последнего обращения)
- **Продвигается** через навык `promote_insight` (продление или снятие срока действия)
- **Отклоняется** через навык `dismiss_insight` (немедленное удаление)
- **Истекает** — задача очистки запускается каждый час и удаляет инсайты после `expires_at`

### Оценка качества инсайтов

После генерации инсайта второй вызов LLM оценивает его:

```
System: "Rate the following insight on a scale of 1-5 for novelty and usefulness."
User: [текст инсайта]
Response: одна цифра 1-5
```

Если `quality_threshold > 0` и оценка ниже порога, инсайт отбрасывается. Это предотвращает засорение памяти низкокачественными инсайтами.

## Эмбеддинги

### ONNX-провайдер

Iulita использует чистый Go ONNX-рантайм (`knights-analytics/hugot`) для локальной генерации эмбеддингов — без внешних API-вызовов.

- **Модель**: `KnightsAnalytics/all-MiniLM-L6-v2` — sentence transformer, 384 измерения
- **Рантайм**: Чистый Go (без CGo, без разделяемых библиотек)
- **Потокобезопасность**: Защита через `sync.Mutex` (пайплайн hugot не потокобезопасен)
- **Кэширование модели**: Загружается один раз в `~/.local/share/iulita/models/`, повторно используется при последующих запусках

### Хранение векторов

Эмбеддинги хранятся как бинарные BLOB в SQLite:

- **Кодирование**: Каждый `float32` -> 4 байта LittleEndian, упакованных в `[]byte`
- **384 измерения** -> 1536 байт на вектор
- **Таблицы**: `fact_vectors` (PK fact_id), `insight_vectors` (PK insight_id)
- **Каскадное удаление**: удаление факта/инсайта автоматически удаляет его вектор

### Кэш эмбеддингов

Таблица `embedding_cache` предотвращает повторное вычисление эмбеддингов для одинаковых текстов:

- **Ключ**: SHA-256 хеш входного текста
- **LRU-вытеснение**: хранит только N последних записей (по умолчанию 10 000)
- **Используется**: обёрткой `CachedEmbeddingProvider` над ONNX

### Алгоритм гибридного поиска

```python
# Псевдокод
def hybrid_search(query, user_id, limit):
    # 1. FTS5-результаты (с избыточной выборкой)
    fts_results = FTS_MATCH(query, limit * 2)
    fts_scores = {r.id: 1 - i/(len+1) for i, r in enumerate(fts_results)}

    # 2. Векторное сходство
    query_vec = onnx.embed(query)
    all_vecs = load_all_vectors(user_id)
    vec_scores = {id: cosine_similarity(query_vec, vec) for id, vec in all_vecs}

    # 3. Комбинирование
    all_ids = set(fts_scores) | set(vec_scores)
    combined = {}
    for id in all_ids:
        fts = fts_scores.get(id, 0)
        vec = vec_scores.get(id, 0)
        combined[id] = (1 - vectorWeight) * fts + vectorWeight * vec

    # 4. Top-N
    return sorted(combined, key=combined.get, reverse=True)[:limit]
```

**Конфигурация**: `skills.memory.vector_weight` (по умолчанию 0, только FTS). Установите 0.3-0.5 для гибридного поиска.

## Память в цикле ассистента

Каждое сообщение инициирует внедрение памяти в системный промпт:

1. **Недавние факты** (до 20): загружаются из БД, применяются затухание + MMR, форматируются как `## Remembered Facts`
2. **Релевантные инсайты** (до 5): гибридный поиск по тексту сообщения, форматируются как `## Insights`
3. **Профиль пользователя** (tech facts): поведенческие метаданные по категориям, форматируются как `## User Profile`
4. **Директива пользователя**: постоянная пользовательская инструкция, форматируется как `## User Directives`

Этот контекст появляется в **динамическом системном промпте** (для каждого сообщения, не кэшируется Claude).

## Экспорт / Импорт памяти

### Экспорт

```go
memory.ExportFacts(ctx, store, chatID) // → строка Markdown
memory.ExportAllFacts(ctx, store, dir) // → один .md файл на чат
```

Формат:
```markdown
## Fact 42
The user prefers dark mode in all IDEs.

## Fact 43
User's favorite programming language is Go.
```

### Импорт

```go
memory.ImportFacts(ctx, store, chatID, markdownContent)
```

Разбирает markdown, создаёт новые факты (исходные ID отбрасываются — новые ID назначаются SQLite autoincrement). Каждый импортированный факт автоматически получает эмбеддинг.

## Справочник конфигурации

| Параметр | По умолчанию | Описание |
|----------|-------------|----------|
| `skills.memory.half_life_days` | 30 | Период полураспада временного затухания; 0 = отключено |
| `skills.memory.mmr_lambda` | 0 | Разнообразие MMR (0 = отключено, 0.7 рекомендуется) |
| `skills.memory.vector_weight` | 0 | Вес гибридного поиска (0 = только FTS, 0.5 = сбалансированный) |
| `skills.insights.min_facts` | 20 | Минимум фактов для генерации инсайтов |
| `skills.insights.max_pairs` | 6 | Максимум межкластерных пар за запуск |
| `skills.insights.ttl` | 720h | TTL инсайта (30 дней) |
| `skills.insights.interval` | 24h | Частота генерации инсайтов |
| `skills.insights.quality_threshold` | 0 | Минимальная оценка качества (0 = принимать все) |
| `embedding.model` | `KnightsAnalytics/all-MiniLM-L6-v2` | Название ONNX-модели |
| `embedding.enabled` | true | Включить ONNX-эмбеддинги |
