# Internationalisation (i18n) et localisation (l10n)

Iulita prend en charge 6 langues a travers toutes les couches : messages systeme backend, console TUI, sorties des competences, vocabulaire d'approbation et le frontend Vue 3 du tableau de bord.

## Langues supportees

| Code | Langue | RTL | Ecriture |
|------|--------|-----|----------|
| `en` | Anglais | Non | Latin |
| `ru` | Russe | Non | Cyrillique |
| `zh` | Chinois (Mandarin) | Non | Hanzi |
| `es` | Espagnol | Non | Latin |
| `fr` | Francais | Non | Latin |
| `he` | Hebreu | Oui | Hebreu |

## Architecture

```
Backend (Go)                          Frontend (Vue 3)
┌─────────────────────┐              ┌─────────────────────┐
│ Paquet i18n         │              │ vue-i18n v9         │
│ ├── Bundle          │              │ ├── Locales JSON    │
│ ├── Catalogues TOML │              │ ├── useLocale()     │
│ └── Mots approbation│              │ └── LanguagePicker  │
└─────────────────────┘              └─────────────────────┘
         ↕                                    ↕
   user_channels.locale              localStorage + PATCH /api/auth/locale
```

## i18n Backend

### Structure du paquet

```
internal/i18n/
├── i18n.go        # Bundle, Init(), T(), Tl(), Translate()
├── context.go     # WithLocale(), LocaleFrom()
├── resolve.go     # ResolveLocale(), SupportedLanguages, matcher
└── catalog/
    ├── en.toml
    ├── ru.toml
    ├── zh.toml
    ├── es.toml
    ├── fr.toml
    └── he.toml
```

### Initialisation

`i18n.Init()` doit etre appele apres les migrations de BDD et avant l'assistant/les competences. Il :

1. Cree un bundle `go-i18n/v2` avec l'anglais comme langue par defaut
2. Enregistre le demarshalleur TOML
3. Lit tous les catalogues `*.toml` embarques depuis `catalog/`
4. Pre-cache les instances `*i18n.Localizer` pour toutes les langues supportees
5. Analyse les mots d'approbation en tables de recherche O(1)

### API de traduction

```go
// Base sur le contexte (lit la locale du contexte)
i18n.T(ctx, "ConsolePlaceholder")
i18n.T(ctx, "WeatherForTitle", map[string]interface{}{"Location": "Moscow"})

// Base sur le tag direct (pas de contexte)
i18n.Tl(language.Russian, "LocaleSwitched", map[string]interface{}{"Language": "Russian"})

// Fonction de traduction principale
i18n.Translate(tag, msgID, data...)
```

Chaine de repli : traduction specifique a la locale → traduction anglaise → msgID brut.

### Propagation du contexte

```go
// Definir la locale dans le contexte
ctx = i18n.WithLocale(ctx, language.Russian)

// Lire la locale du contexte
tag := i18n.LocaleFrom(ctx) // retourne language.English si non defini
```

La locale circule a travers tout le cycle de vie de la requete :

```
user_channels.locale (BDD)
    ↓
msg.Locale (champ IncomingMessage)
    ↓
i18n.ResolveLocale(msg.Locale, msg.LanguageCode)
    ↓
i18n.WithLocale(ctx, tag)
    ↓
Tous les appels i18n.T(ctx, ...) en aval
```

### Resolution de la locale

`ResolveLocale(channelLocale, msgLang)` :

1. Analyser `channelLocale` (preference stockee en BDD, ex. `"ru"`)
2. Se replier sur `msgLang` (langue de la plateforme, ex. `LanguageCode` de Telegram)
3. Passer par `language.NewMatcher(SupportedLanguages)` pour la meilleure correspondance
4. **Priorite** : locale stockee en BDD > langue de la plateforme > anglais par defaut

### Cles du catalogue

Les 6 locales couvrent ces groupes de cles :

| Groupe | Nombre | Exemples |
|--------|--------|----------|
| Flux d'approbation | 9 | `ApprovalAffirmative`, `ApprovalNegative`, `ApprovalPrompt` |
| Telegram | 7 | `TelegramHistoryCleared`, `TelegramRateLimited` |
| Console TUI | 17 | `ConsolePlaceholder`, `ConsoleThinking`, `ConsoleProcessing` |
| Commandes console | 12 | `CommandHelpHeader`, `CommandStatusSkills` |
| Meteo | 30+ | Codes meteo OMM, `WeatherLocationPrompt` |
| Competence locale | 3 | `LocaleSwitched`, `LocaleUnknown` |
| Assistant | 3 | `AssistantLanguageDirective`, `AssistantCompressionPrompt` |
| Erreur | 1 | `ErrorUnknownSkill` |

### Mots d'approbation

Chaque locale definit des mots affirmatifs et negatifs pour le flux d'approbation :

```toml
# ru.toml
[ApprovalAffirmative]
other = "да,д,ок,подтвердить,подтверждаю,yes,y,ok,confirm"

[ApprovalNegative]
other = "нет,н,отмена,отменить,no,n,cancel"
```

**Garantie inter-langues** : toutes les locales non anglaises incluent les mots anglais en repli. Les utilisateurs peuvent taper "yes" quelle que soit leur locale.

La verification d'approbation est en O(1) grace a des ensembles de mots pre-analyses (`map[string]map[string]bool`).

### Directive de langue LLM

Lorsque la locale n'est pas l'anglais, une directive de langue est injectee dans le prompt systeme dynamique :

```
# Russe :
"Отвечай на русском языке. Язык ответов должен соответствовать предпочтениям пользователя."

# Hebreu :
"השב בעברית. שפת התשובות חייבת להתאים להעדפת המשתמש."
```

La directive est rendue dans la langue cible pour que le LLM voie des instructions correspondant a sa langue de reponse attendue. Les en-tetes de section (`## Current Time`, `## Remembered Facts`) restent en anglais — le LLM les comprend quel que soit le contexte.

### Notes sur les catalogues TOML

- **Guillemets chinois** : utiliser `「」` au lieu de `""` dans `zh.toml` — l'analyseur TOML ne gere pas les guillemets pleine largeur dans les chaines ordinaires
- **Prompt de compression** : conserve en anglais dans toutes les locales — le LLM interprete correctement les instructions de resume en anglais
- **Prefixe de resume** : traduit (apparait directement dans le chat)

## Stockage de la locale par canal

La locale est stockee par canal, pas par utilisateur :

```sql
user_channels (
    ...
    locale TEXT NOT NULL DEFAULT 'en'
)
```

Cela permet des langues differentes sur differents canaux (ex. russe sur Telegram, anglais sur WebChat).

### Operations de stockage

- `UpdateChannelLocale(ctx, chatID, locale)` — appele par la competence `set_language`
- `GetChannelLocale(ctx, channelType, channelUserID)` — appele lors du traitement des messages
- `GetChannelLocaleByChatID(ctx, chatID)` — recherche alternative

### Competence set_language

Les utilisateurs changent de langue via le chat :

```
Utilisateur : "switch to Russian"
Bot : "Язык интерфейса изменён на русский."  (en russe)

Utilisateur : "переключи на иврит"
Bot : "שפת הממשק שונתה לעברית."  (en hebreu)
```

La competence :
1. Analyse l'entree de langue (code BCP-47 ou nom en anglais/russe)
2. Valide contre les langues supportees
3. Met a jour `user_channels.locale` en BDD
4. Retourne la confirmation dans la **nouvelle** langue

## i18n Frontend

### Configuration

```typescript
// ui/src/i18n/index.ts
const i18n = createI18n({
    legacy: false,  // Mode Composition API
    locale: localStorage.getItem('iulita.locale') || navigator.language.split('-')[0] || 'en',
    fallbackLocale: 'en',
    messages: { en, ru, zh, es, fr, he }
})
```

Les 6 fichiers de locale JSON sont importes statiquement et inclus dans le bundle a la compilation.

### Composable useLocale

```typescript
// ui/src/composables/useLocale.ts
const { currentLocale, isRTL, direction, setLocale } = useLocale()
```

- `currentLocale` — ref reactive, partagee entre toutes les instances du composable
- `isRTL` — computed : `true` pour l'hebreu
- `direction` — computed : `'rtl'` ou `'ltr'`
- `setLocale(tag)` :
  1. Met a jour la ref `currentLocale`
  2. Definit `locale.value` (vue-i18n)
  3. Sauvegarde dans `localStorage`
  4. Definit `document.documentElement.lang`
  5. `PATCH /api/auth/locale` en best-effort pour synchroniser le backend

### Support RTL

Le RTL est applique par composant (pas sur `<html>`) pour eviter de casser la mise en page de la barre laterale. Les composants utilisent la valeur computed `direction` pour definir les attributs `dir` la ou necessaire.

### Composant LanguagePicker

Un menu deroulant Naive UI `<NSelect>` dans la barre laterale avec 6 options libellees dans leurs ecritures natives (ex. "עברית" pour l'hebreu). A la selection, appelle `setLocale()`.

### Structure des locales JSON

Objets d'espaces de noms imbriques :

```json
{
  "nav": { "dashboard": "Dashboard", "facts": "Facts", ... },
  "common": { "save": "Save", "cancel": "Cancel", ... },
  "login": { "title": "Login", ... },
  "dashboard": { ... },
  "tasks": { ... },
  "settings": { ... },
  "locale": { "en": "English", "ru": "Русский", "he": "עברית", ... }
}
```

### Synchronisation backend de la locale

`PATCH /api/auth/locale` (endpoint du tableau de bord) :
1. Valide la locale via `i18n.IsSupported()`
2. Met a jour `user_channels.locale` pour tous les canaux de l'utilisateur
3. Retourne le profil utilisateur mis a jour

## Ajouter une nouvelle langue

### Backend

1. Creer `internal/i18n/catalog/<code>.toml` avec toutes les cles requises
2. Ajouter le `language.Tag` a `SupportedLanguages` dans `resolve.go`
3. Ajouter le drapeau RTL si necessaire
4. Ajouter les correspondances de noms de langue dans `skill/locale/locale.go` (noms anglais + russe)

### Frontend

1. Creer `ui/src/i18n/locales/<code>.json` avec tous les espaces de noms
2. Importer et enregistrer dans `ui/src/i18n/index.ts`
3. Ajouter a RTL_LOCALES dans `ui/src/composables/useLocale.ts` si RTL
4. Ajouter le libelle natif dans tous les fichiers de locale sous `locale.<code>`
